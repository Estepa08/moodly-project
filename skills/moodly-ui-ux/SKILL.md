# UI/UX для Moodly

## Контекст продукта

Moodly — трекер ментального здоровья. Цель: дать человеку удобно отслеживать параметры состояния (тревожность, сон, настроение), проходить психологические тесты с расчётом балла и рекомендацией, и превратить работу над своим мышлением в приятную привычку. Три стратегии продукта: удобное отслеживание, профессиональные рекомендации, приятная привычка (не давление).

Полный список сущностей и API — в `api-contract/` репозитория (User, Parameter, Entry, Test/TestResult, Feedback, OnboardingStory, Report). Экраны проектируются под эти сущности, не придумывай новые данные без необходимости — сначала проверь, укладывается ли задача в существующую модель.

## Принципы дизайна и почему они важны именно здесь

**Спокойствие важнее вовлечения.** Это не соцсеть и не фитнес-трекер с гонкой за очками — это приложение о ментальном здоровье. Механики, которые давят на пользователя (агрессивные пуш-уведомления, счётчики "ты пропустил N дней", красные предупреждающие бейджи за бездействие) могут усиливать тревожность у самой уязвимой аудитории продукта. В плане это уже заложено: "Не бесячий интерфейс — цвета, шрифты, анимация — плавная" входит в MVP не просто как красивость, а как требование к продукту.

**Цвет никогда не единственный носитель смысла.** В бизнес-плане явно записано "визуальные индикаторы через цвет и иконки" — не только цвет. Причина не только в стиле: часть пользователей не различает цвета (примерно 1 из 12 мужчин), и если состояние параметра кодируется только оттенком, для них это не читается. Всегда дублируй цвет иконкой, формой или подписью.

**Прогресс — а не сравнение.** Стрики, квесты и достижения (эти механики отмечены в бизнес-плане как "хотелось бы, но продукт может обойтись" — не MVP) должны показывать личный прогресс пользователя, а не сравнение с другими или давление продолжать серию любой ценой. Не проектируй "разрыв стрика" как наказание с потерей данных — это раздел, где легко случайно создать источник тревоги вместо инструмента заботы.

**Формулировки результатов тестов — рекомендательные, не диагностические.** Экран результата TestResult показывает балл и рекомендацию, но это не заменяет консультацию специалиста. Даже при "тревожных" результатах тон должен оставаться поддерживающим, а не пугающим, с явным указанием обратиться к специалисту при необходимости — но без драматизации.

## Экраны и потоки

Ориентируйся на приоритет MVP → должно быть → хотелось бы из бизнес-плана — не проектируй хотелось-бы-фичи (свой "питомец", биографии, лента отзывов) раньше, чем закрыты базовые потоки.

**Онбординг (OnboardingStory, MVP)** — короткая последовательность экранов сразу после регистрации, знакомит с тем, что делать в приложении. Это контент, не интерактивная механика — приоритет здесь на текст/иллюстрацию, а не на сложную анимацию.

**Трекинг (Parameter + Entry, MVP)** — быстрый ввод значения параметра должен занимать секунды, не экран с формой из 10 полей. Разделяй "быстрый ввод" (главный экран, для ежедневного использования) и "детальный вид" (история записей, графики) — это прямое требование из плана: "расставить акцент на ключевых параметрах с возможностью провалиться в детали".

**Тесты (Test + TestResult, MVP/должно быть)** — по умолчанию один вопрос на экран с прогресс-баром, а не длинная форма целиком: так пользователь не видит объём заранее и не откладывает прохождение из-за "это долго". Экран результата — балл, понятная интерпретация простыми словами, рекомендация, без списка медицинских терминов без объяснений.

**Отчёт для врача (Report, должно быть)** — выбор периода и формата (PDF/CSV), кнопка "скачать". Врач не пользователь приложения — экран проектируется для того, кто отчёт создаёт, а не для получателя.

**Геймификация (должно быть/хотелось бы — не блокирует MVP)** — форматы тренировок кроме свайпа (квест и т.п.), "цитата дня" как мини-сторис, анимация вовлечения — всё это второй слой поверх работающего MVP. Если тебя просят спроектировать это раньше базовых потоков — сначала уточни, действительно ли MVP уже закрыт.

## Тон и голос бренда

Название продукта — Moodly (лёгкое, дружелюбное, в духе Meetly/Calendly, не клиническое). Визуальный и текстовый тон должен соответствовать: тёплый, простой язык, без канцеляризмов и без медицинской строгости, но без инфантилизации серьёзной темы.

## Чек-лист доступности

- Контраст текста — минимум AA (4.5:1 для основного текста).
- Интерактивные элементы — не мельче 44×44px для тапа.
- Иконки, несущие смысл (не декоративные), должны иметь текстовую альтернативу для скринридеров.
- Анимации — с возможностью отключить/уменьшить (respect `prefers-reduced-motion`), это особенно важно для пользователей с тревожными расстройствами, которых может раздражать избыточное движение на экране.

## Если неясно, что в скоупе

Свериться с приоритизацией из бизнес-плана (MVP / должно быть / хотелось бы) и с AGENTS.md в корне репозитория — там зафиксировано, какие сущности read-only, а какие пишет пользователь.
